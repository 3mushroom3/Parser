# Запуск и настройка FSA Parser

## Быстрый старт

```powershell
cd backend
npm install          # один раз
node server.js       # запуск
```

Сервер поднимается на `http://localhost:3001`.  
Через 2 секунды после старта автоматически запускается первый парсинг.

---

## Переменные окружения

Все переменные имеют дефолтные значения — без них сервер тоже запустится.

### Фильтрация реестра FSA

| Переменная | Пример | Описание |
|---|---|---|
| `FSA_TECH_REG_IDS` | `32` | **Числовой ID технического регламента** — передаётся в `filter.idTechReg`. Самый надёжный способ фильтрации. `32` = «О безопасности зерна» (ТР ТС 015/2011). Несколько: `32,15`. |
| `FSA_TECH_REGLAMENT` | `о безопасности зерна` | Локальный фильтр: подстрока **полного названия** регламента из ответа API. Работает **после** получения списка, т.е. не сокращает число запросов к FSA, зато точнее чем текстовый поиск. |
| `FSA_DATE_FROM` | `2024-01-01` | Дата начала регистрации (YYYY-MM-DD). Передаётся в `filter.regDate.minDate`. |
| `FSA_DATE_TO` | `2024-12-31` | Дата конца регистрации (YYYY-MM-DD). Передаётся в `filter.regDate.maxDate`. |
| `FSA_DATE_CHUNK` | `week` | **Нарезка на окна для полного скачивания.** Значения: `week` (7 дн.), `biweek` (14), `month` (30), `day` (1) или число дней. Требует `FSA_DATE_FROM`. Обходит жёсткий лимит FSA (21 стр./сессию). |
| `FSA_FILTERS` | `{"status":[1]}` | Произвольные поля в `filter{}` FSA API (JSON). Перезаписывают базовые поля. |

> **Важно:** FSA API хранит регламент как **полное название** («О безопасности зерна»), а не код («ТР ТС 015/2011»). Поиск по коду не работает.

### Лимиты и производительность

| Переменная | Дефолт | Описание |
|---|---|---|
| `FSA_PAGE_SIZE` | `100` | Размер страницы (максимум 100). |
| `FSA_MAX_PAGES_PER_RUN` | `0` | Максимум страниц за один прогон. `0` = без лимита (качать всё). |
| `FSA_MAX_RECORDS` | `0` | Максимум записей в локальной БД. `0` = без лимита. После каждого прогона БД обрезается. |
| `FSA_FETCH_CONCURRENCY` | `3` | Параллельных запросов карточек деклараций. |
| `FSA_DELAY_MS` | `1500` | Задержка между страницами (мс). Увеличьте при частых 503. |
| `FSA_CRON_SCHEDULE` | `*/30 * * * *` | Расписание автоматического парсинга (cron). |

### Таймауты HTTP

| Переменная | Дефолт | Описание |
|---|---|---|
| `FSA_HTTP_TIMEOUT_MS` | `45000` | Таймаут API-запросов (мс). |
| `FSA_LOGIN_TIMEOUT_MS` | `60000` | Таймаут POST /login (мс). |
| `FSA_BOOTSTRAP_TIMEOUT_MS` | `90000` | Таймаут загрузки стартовой страницы (мс). |

### Авторизация

| Переменная | Описание |
|---|---|
| `FSA_MANUAL_TOKEN` | JWT-токен вручную — минует авто-логин. Полезен при нестабильной авторизации FSA. |

Токен также можно установить в runtime через API (без перезапуска):
```
POST http://localhost:3001/api/settoken
Body: { "token": "eyJ..." }
```

### Прочее

| Переменная | Дефолт | Описание |
|---|---|---|
| `PORT` | `3001` | Порт HTTP-сервера. |
| `FSA_BASE_URL` | `https://pub.fsa.gov.ru` | Базовый URL FSA. |

## Лимит FSA API: 21 страница (~2100 записей) на один запрос

FSA API жёстко ограничивает пагинацию: страница 21+ возвращает ошибку `RDS-APP-9995`.  
При этом `total` в ответе **всегда показывает 1000** — это заглушка, реальное число записей неизвестно.

**Реальные цифры:** даже один месяц содержит >2100 деклараций по зерновому регламенту.  
Без `FSA_DATE_CHUNK` парсер получает только **первые 2100 (новейших)**, остальные теряются.

### Полное скачивание через FSA_DATE_CHUNK=week

```powershell
# Полное скачивание 2024 года по неделям (~52 окна × ~490 записей = все записи)
$env:FSA_TECH_REG_IDS="32"
$env:FSA_DATE_FROM="2024-01-01"
$env:FSA_DATE_TO="2024-12-31"
$env:FSA_DATE_CHUNK="week"
node server.js
```

```powershell
# Скачать все годы 2019–2025 полностью (долго — тысячи записей)
$env:FSA_TECH_REG_IDS="32"
$env:FSA_DATE_FROM="2019-01-01"
$env:FSA_DATE_TO="2025-12-31"
$env:FSA_DATE_CHUNK="week"
node server.js
```

Каждый запуск **дополняет** БД, дубликаты не добавляются. Можно прерывать и продолжать.

## Типичные сценарии запуска

### Полное скачивание зерновых за конкретный год (рекомендуется)
```powershell
$env:FSA_TECH_REG_IDS="32"
$env:FSA_DATE_FROM="2025-01-01"
$env:FSA_DATE_TO="2025-12-31"
$env:FSA_DATE_CHUNK="week"
node server.js
```

### Только свежие декларации (последние ~2100 — быстро, без chunking)
```powershell
$env:FSA_TECH_REG_IDS="32"; $env:FSA_DATE_FROM="2025-01-01"; node server.js
```

### Полный реестр без фильтров (получит только первые ~2100 записей)
```powershell
node server.js
```

### Активные декларации по зерну
```powershell
$env:FSA_TECH_REG_IDS="32"; $env:FSA_FILTERS='{"status":[1]}'; $env:FSA_DATE_CHUNK="week"; $env:FSA_DATE_FROM="2024-01-01"; node server.js
```

### Ограниченный прогон для теста (только первые 5 страниц окна)
```powershell
$env:FSA_MAX_PAGES_PER_RUN="5"; node server.js
```

### При нестабильном FSA (частые 503 — увеличить задержку)
```powershell
$env:FSA_DELAY_MS="5000"; node server.js
```

## Ошибки и как с ними работать

| Ошибка | Причина | Решение |
|---|---|---|
| `HTTP 503` | FSA временно недоступен | Парсер автоматически делает до 10 повторов с экспоненциальной задержкой. Ничего делать не нужно. |
| `RDS-APP-9995` (HTTP 400) | Превышен лимит 21 страницы на запрос | Включить `FSA_DATE_CHUNK=week` + задать `FSA_DATE_FROM`. Если уже используется — уменьшить chunk (попробовать `day`). |
| `POST /login: статус 403` | Нет cookie XSRF-TOKEN | Сервер FSA не отдал CSRF-токен — проблема с сетью или прокси. Попробовать позже или задать `FSA_MANUAL_TOKEN`. |
| `Не удалось получить токен` | Авторизация упала полностью | Задать `FSA_MANUAL_TOKEN` (взять из браузера: DevTools → Network → заголовок `Authorization`). |

## Как найти ID технического регламента

```powershell
cd backend
node scripts/probe-api.js --tech-regs зерна
```

Скрипт переберёт ID 1–100, выведет совпадения.

Известные ID:
- `32` — «О безопасности зерна» (ТР ТС 015/2011)

## Диагностика

```powershell
# Проверить авторизацию и первые 2 декларации из реестра
node scripts/probe-api.js

# Проверить конкретную декларацию по ID
node scripts/probe-api.js 16827658

# HTTP-диагностика через API сервера (должен быть запущен)
curl http://localhost:3001/api/debug/fsa

# Статус парсера
curl http://localhost:3001/api/status
```

## Автоматический запуск (cron)

По умолчанию парсер запускается **каждые 30 минут** (`*/30 * * * *`).

Изменить расписание:
```powershell
$env:FSA_CRON_SCHEDULE="0 */6 * * *"   # каждые 6 часов
$env:FSA_CRON_SCHEDULE="0 2 * * *"     # каждый день в 02:00
node server.js
```

Запустить парсинг вручную (без перезапуска сервера):
```
POST http://localhost:3001/api/parse
```

## Данные

- **БД:** `data/declarations.json` — JSON-массив, новые записи добавляются в начало.
- **Статус:** `data/status.json` — последний результат прогона.
- **Экспорт CSV:** `GET http://localhost:3001/api/export/csv` (UTF-8 BOM, открывается в Excel).
- Существующие записи при повторном прогоне **не дублируются** — только обновляется статус.
